## Purpose

This document defines the canonical structure and standards for all workspaces in the Nixtla plugins repository. Every workspace (timegpt-lab, statsforecast-lab, mlforecast-lab, neuralforecast-lab, hierarchicalforecast-lab) must conform to these standards for consistency, automation, and developer experience.

## Directory Standards

### skills/

**Purpose**: Claude Skills (SKILL.md files) specific to this Nixtla domain.

**Rules**:
- Every skill must be a directory with `SKILL.md` at root
- Follow master SKILL.md standard: `000-docs/077-SPEC-MASTER-claude-skills-standard.md`
- Use `{baseDir}/` for all file paths (never absolute paths)
- Descriptions must be third-person, imperative voice
- Skills here are **prototypes** - promote to `003-skills/` when stable

### scripts/

**Purpose**: Python scripts for experiments, data processing, model training, evaluation.

**Rules**:
- All scripts must be executable Python 3.10+ (or compatible with repo's Python version)
- Use requirements.txt for dependencies (install in workspace venv)
- Scripts should be runnable from workspace root: `python scripts/my_script.py`
- Use argparse for CLI arguments (no hardcoded paths)
- Log output to stdout/stderr (scripts/ is for experimentation, not production logging)

### data/

**Purpose**: Sample datasets, experiment data, API response caches, model inputs.

**Rules**:
- Use .gitignore for large files (> 10MB should not be committed)
- Organize by experiment or dataset name: `data/m4_daily_small/`, `data/api_cache/`
- Include README.md if datasets need documentation
- Use standard Nixtla schema: `unique_id`, `ds`, `y` (where applicable)
- Archive old experiments to `data/010-archive/` (or delete if no longer needed)

## Workspace README Requirements

Every workspace must have a README.md with these sections:

1. **Title** (e.g., `# TimeGPT Lab`)
2. **Purpose** (2-3 sentences explaining domain and experimentation focus)
3. **Structure** (brief bullets for skills/, scripts/, data/, reports/, docs/)
4. **Example Future Flows** (5+ numbered examples of typical workflows)
5. **Environment Setup** (pip install commands, API keys, env vars)
6. **Promotion Path** (when to promote to skills-pack or plugins)

## Naming Conventions

- **Workspaces**: `{domain}-lab` (all lowercase, kebab-case)
  - Valid: `timegpt-lab`, `statsforecast-lab`, `mlforecast-lab`
  - Invalid: `TimeGPT_Lab`, `statsForecastLab`, `ML-Forecast-Lab`

- **Skills**: `nixtla-{domain}-{function}` (all lowercase, kebab-case)
  - Valid: `nixtla-timegpt-forecaster`, `nixtla-statsforecast-baseline`
  - Invalid: `TimeGPT_Forecaster`, `nixtla_stats_forecast`

- **Scripts**: `{function}_{context}.py` (lowercase, snake_case)
  - Valid: `forecast_sample.py`, `run_m4_benchmark.py`, `compute_ig.py`
  - Invalid: `ForecastSample.py`, `run-m4-benchmark.py`, `script1.py`

## Promotion Flow

Workspaces are **internal labs** - code graduates to production/distribution layers:

1. **Prototype in workspace**: Design, experiment, validate in `{workspace}/skills/` or `{workspace}/scripts/`
2. **Validate**: Test thoroughly, document, ensure reproducibility
3. **Promote**:
   - **Skills**: Move to `003-skills/.claude/skills/nixtla-{domain}-{function}/`
   - **Scripts**: Integrate into `005-plugins/nixtla-{domain}-*/scripts/`
   - **Docs**: Extract to `000-docs/` with proper NNN-CC-ABCD naming
4. **Update metadata**: Update plugin manifest, skills index, CHANGELOG.md
5. **Release**: Tag version, create release AAR in `000-docs/`

## CI/CD Integration

Each workspace can have:
- **Smoke tests**: `{workspace}/tests/` (run in CI to validate workspace setup)
- **Golden tasks**: Representative experiments that must pass before promotion
- **Environment validation**: Scripts to check API keys, dependencies, data availability

CI workflows in `.github/workflows/` can target specific workspaces for validation.

## Version Control

- **Do commit**: Skills (SKILL.md), scripts, small datasets (< 10MB), docs, reports (< 1MB)
- **Do not commit**: Large datasets (> 10MB), model checkpoints, API caches (use .gitignore)
- **Archive strategy**: Move old experiments to `{dir}/archive/` or delete if no longer needed

## Standards Compliance

All workspaces must comply with:
- **Master SKILL.md Standard**: `000-docs/077-SPEC-MASTER-claude-skills-standard.md`
- **Claude Skills Frontmatter Schema**: `000-docs/6767-m-DR-STND-claude-skills-frontmatter-schema.md`
- **Claude Skills Authoring Guide**: `000-docs/6767-n-DR-GUID-claude-skills-authoring-guide.md`
- **Doc-Filing v4.2**: `000-docs/` flat structure with NNN-CC-ABCD naming

## Violations

If a workspace violates these standards:
1. Create issue in GitHub with label `standards-violation`
2. Document violation in `000-docs/` with AA-STAT type code
3. Fix violation or update standards (with proper AAR)
4. Re-validate with CI
