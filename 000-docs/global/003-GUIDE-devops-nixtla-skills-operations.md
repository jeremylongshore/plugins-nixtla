## Introduction: What Are We Talking About?

Hey there! If you're reading this, someone on your team probably wants to use **Nixtla Claude Skills** in their forecasting projects, and you're the person who needs to understand how these things work from an operations perspective.

Don't worry - this isn't complicated. But it's different from traditional tools, so let's take 15 minutes to walk through what these skills are, how they're installed, and what you need to know to keep your team happy and your infrastructure secure.

---

## Part 1: What Are Nixtla Skills? (3 minutes)

### The 10-Second Explanation

Nixtla Skills are **AI assistant modes** for Claude Code that help data scientists build time series forecasting workflows faster. Think of them like specialized plugins that make Claude Code smarter about forecasting.

### The More Complete Explanation

Here's the deal:

1. **Claude Code** is an AI coding assistant (like GitHub Copilot, but Claude)
2. **Skills** are special instruction files that teach Claude how to do specific tasks
3. **Nixtla Skills** are a collection of 6 skills specifically for time series forecasting

When a data scientist uses these skills, Claude can:
- Set up forecasting experiments automatically
- Compare different forecasting models
- Generate production-ready pipeline code
- Audit API usage and suggest cost optimizations
- And more...

### Where Do Skills Live?

Skills are just **markdown files** stored in `.claude/skills/` directories within a project:

```
my-forecasting-project/
├── .claude/
│   └── skills/
│       ├── nixtla-timegpt-lab/
│       │   └── SKILL.md                  # ← The skill definition
│       ├── nixtla-schema-mapper/
│       │   └── SKILL.md
│       ├── nixtla-experiment-architect/
│       │   └── SKILL.md
│       ├── nixtla-timegpt-finetune-lab/
│       │   └── SKILL.md
│       ├── nixtla-prod-pipeline-generator/
│       │   └── SKILL.md
│       └── nixtla-usage-optimizer/
│           └── SKILL.md
├── forecasting/
│   └── ... (code generated by skills)
└── data/
    └── ... (time series data)
```

**Key point**: Skills are **per-project**, not global. Each project that wants to use Nixtla skills needs its own copy.

## Part 2: The Skills Lifecycle (5 minutes)

Let's walk through how skills move through your infrastructure, from installation to eventual removal.

### Step 1: Installation

**Who does this?** Usually the data scientist or ML engineer.

**What happens?** They run an installer that copies skill files into their project:

```bash
# Navigate to their project
cd my-forecasting-project/

# Install Nixtla skills (hypothetical installer - adjust when released)
pip install nixtla-claude-skills-installer
nixtla-skills init

# Skills are now in .claude/skills/
ls .claude/skills/
# → nixtla-timegpt-lab/
# → nixtla-schema-mapper/
# → ... (all 6 skills)
```

**What did that do?**

- Copied 6 skill directories (each with a SKILL.md file) to `.claude/skills/`
- Created a `.claude/skills.json` manifest (metadata about installed skills)
- That's it! No system-wide changes, no daemon processes, nothing global.

### Step 2: Discovery (How Claude Finds Skills)

**When the data scientist opens this project in Claude Code:**

1. Claude Code automatically scans `.claude/skills/` directories
2. Reads the SKILL.md files
3. Makes those skills available when the user activates them

**Key point for DevOps**: Skills are **passive files**. They don't run code on installation. They're just instructions Claude reads when needed.

### Step 3: Usage

**When a data scientist uses a skill:**

```
# In Claude Code, they type:
"Use nixtla-timegpt-lab to run a forecast on my data"
```

**What happens?**

1. Claude reads the SKILL.md file (instructions)
2. Asks the user questions (dataset path, horizon, etc.)
3. Generates code files (Python scripts, config files)
4. User runs the generated code manually or via automation

**Important**: The skill doesn't execute code directly. It **generates** code that the user then runs.

### Step 4: Updates

**When skills need updating** (bug fixes, new features):

```bash
# Update the installer package
pip install --upgrade nixtla-claude-skills-installer

# Re-run init to update skills
nixtla-skills init --update
```

**What this does:**
- Overwrites existing skill files with new versions
- Preserves user data (forecasting results, configs, etc.)
- Updates the `.claude/skills.json` manifest

**Rollback strategy:**
- Skills are versioned (version in SKILL.md frontmatter)
- You can git commit before updating, then revert if needed
- Or manually save `.claude/skills/` before updating

### Step 5: Removal

**To remove skills from a project:**

```bash
# Manual removal
rm -rf .claude/skills/nixtla-*

# Or use installer (if it provides uninstall)
nixtla-skills remove
```

**What this does:**
- Deletes skill files
- User-generated code (forecasting/, pipelines/) is **not affected**
- Data scientist can still use the code they generated, just can't regenerate it

## Part 3: Security and Configuration (4 minutes)

Okay, now let's talk about the stuff that keeps you up at night: security, credentials, and API keys.

### API Keys: The NIXTLA_API_KEY

Many Nixtla skills use **TimeGPT**, which is a paid API. This requires an API key.

**How users set it:**

```bash
# As an environment variable
export NIXTLA_API_KEY='sk-proj-abc123...'

# Or in a .env file (gitignored)
echo 'NIXTLA_API_KEY=sk-proj-abc123...' >> .env
```

**DevOps best practices:**

1. ✅ **DO**: Use environment variables or secret managers (AWS Secrets Manager, GCP Secret Manager, etc.)
2. ✅ **DO**: Add `.env` to `.gitignore` (prevent committing keys)
3. ✅ **DO**: Rotate keys regularly (every 90 days)
4. ✅ **DO**: Use different keys for dev, staging, prod
5. ❌ **DON'T**: Commit keys to git repositories
6. ❌ **DON'T**: Share keys via Slack, email, or other insecure channels
7. ❌ **DON'T**: Use production keys in local development

**Example gitignore:**

```gitignore
# Environment variables
.env
.env.local
.env.*.local

# API keys
*.key
secrets/

# Claude Code installed skills (optional - usually keep in git)
# .claude/
```

### Network Access

Skills may generate code that makes API calls to:

- **Nixtla TimeGPT API**: `api.nixtla.io` (HTTPS)
- **Data sources**: PostgreSQL, BigQuery, S3, etc. (depends on your setup)
- **External packages**: PyPI (for pip installs)

**Firewall considerations:**

- If your infrastructure has strict egress rules, whitelist:
  - `api.nixtla.io` (TimeGPT API)
  - `pypi.org` (for pip installs)
  - Your data warehouse/database endpoints

**Proxy support:**

If you use a corporate proxy, ensure Python can reach it:

```bash
export HTTP_PROXY='http://proxy.company.com:8080'
export HTTPS_PROXY='http://proxy.company.com:8080'
```

### Data Privacy

**What data do skills see?**

- Skills **do not send your data to Nixtla** unless the user explicitly calls TimeGPT API
- Skills read local files to understand schema, but don't transmit them
- Only when the user runs `client.forecast(df=data)` is data sent to TimeGPT API

**GDPR/Compliance considerations:**

- If your data is sensitive (PII, financial), ensure:
  - Data is anonymized before forecasting (remove identifiers)
  - Nixtla's terms of service meet your compliance requirements
  - API keys are scoped to non-production data in dev environments

### Dependency Management

Skills generate code that requires Python packages:

```bash
pip install nixtla statsforecast mlforecast
```

**DevOps considerations:**

1. **Dependency pinning**: Pin versions in `requirements.txt` or `pyproject.toml`
   ```
   nixtla==0.5.2
   statsforecast==1.7.0
   mlforecast==0.11.0
   ```

2. **Virtual environments**: Always use venvs to isolate dependencies
   ```bash
   python3 -m venv venv
   source venv/bin/activate
   pip install -r requirements.txt
   ```

3. **Container isolation**: For production, use Docker to lock dependencies
   ```dockerfile
   FROM python:3.10-slim
   COPY requirements.txt .
   RUN pip install --no-cache-dir -r requirements.txt
   COPY . /app
   CMD ["python", "forecasting/experiments.py"]
   ```

## Part 4: Recommended DevOps Practices (5 minutes)

Now that you understand how skills work, here's how to set up your team for success.

### Practice 1: Per-Project Installation (Not Global)

**Do this:**
```bash
# Each project has its own skills
cd project-A && nixtla-skills init
cd project-B && nixtla-skills init
```

**Don't do this:**
```bash
# ❌ Don't try to install skills globally
sudo nixtla-skills install --global  # This doesn't exist and shouldn't
```

**Why?**
- Different projects may need different skill versions
- Per-project isolation prevents conflicts
- Easier to audit what's using what

### Practice 2: Version Control for Skills

**Add skills to git** (optional, but recommended):

```bash
git add .claude/skills/
git commit -m "chore: add Nixtla skills v1.0.0"
```

**Why?**
- Team members get the same skill versions automatically
- Easy to see what changed between versions (git diff)
- Rollback is simple (git revert)

**Alternative**: If skills are large, add to `.gitignore` and document installation in README:

```markdown
## Setup

1. Clone this repo
2. Install skills: `nixtla-skills init`
3. Install dependencies: `pip install -r requirements.txt`
```

### Practice 3: Separate Dev, Staging, Prod API Keys

**Use different keys per environment:**

```bash
# Development
export NIXTLA_API_KEY='sk-dev-abc123...'

# Staging
export NIXTLA_API_KEY='sk-staging-def456...'

# Production
export NIXTLA_API_KEY='sk-prod-ghi789...'
```

**Store in your secret manager:**

- AWS: Use AWS Secrets Manager, inject via ECS task definitions
- GCP: Use Secret Manager, inject via Cloud Run environment variables
- Azure: Use Key Vault, inject via App Service configuration
- Kubernetes: Use Kubernetes Secrets, mount as env vars

**Example (GCP Cloud Run):**

```bash
gcloud run deploy forecasting-service \
  --image gcr.io/my-project/forecast:latest \
  --set-env-vars "NIXTLA_API_KEY=secretmanager://projects/my-project/secrets/nixtla-api-key"
```

### Practice 4: Monitor API Usage and Costs

**Set up monitoring for:**

1. **API call volume**: How many TimeGPT requests per day/week/month?
2. **API costs**: What's the monthly bill? (Nixtla provides usage dashboards)
3. **Error rates**: Are API calls failing? (indicates key issues or outages)

**Implement alerts:**

```yaml
# Example alert (pseudocode)
alert: nixtla_api_cost_spike
condition: daily_cost > $100
action: notify_devops_team
```

**Use the usage-optimizer skill** to audit and reduce costs:

```
# Have your data scientist run this periodically
"Use nixtla-usage-optimizer to analyze our TimeGPT usage"
```

This generates a report showing where you're spending money and how to optimize.

### Practice 5: CI/CD Integration

**For teams deploying forecast pipelines to production:**

1. **Lint generated code** (ensure skills generate valid Python)
   ```bash
   flake8 forecasting/ pipelines/
   ```

2. **Test generated code** (run unit tests on generated scripts)
   ```bash
   pytest tests/
   ```

3. **Validate forecasts** (backtesting, sanity checks)
   ```bash
   python forecasting/experiments.py --validate
   ```

4. **Deploy with Docker** (containerize for consistency)
   ```dockerfile
   FROM python:3.10-slim
   WORKDIR /app
   COPY requirements.txt .
   RUN pip install --no-cache-dir -r requirements.txt
   COPY forecasting/ ./forecasting/
   COPY pipelines/ ./pipelines/
   CMD ["python", "pipelines/timegpt_forecast_dag.py"]
   ```

**Example CI pipeline** (GitHub Actions):

```yaml
name: Forecast Pipeline CI

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with:
          python-version: '3.10'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest flake8

      - name: Lint
        run: flake8 forecasting/ pipelines/

      - name: Test
        run: pytest tests/

      - name: Validate forecasts
        env:
          NIXTLA_API_KEY: ${{ secrets.NIXTLA_API_KEY_DEV }}
        run: python forecasting/experiments.py --validate
```

## Part 5: Troubleshooting Common Issues (3 minutes)

Here are the problems you'll probably encounter and how to fix them.

### Issue 1: "NIXTLA_API_KEY not set"

**Symptom**: User runs forecasting code, gets error: `NIXTLA_API_KEY environment variable not set`

**Solution**:
```bash
# Check if key is set
echo $NIXTLA_API_KEY

# If empty, set it
export NIXTLA_API_KEY='sk-proj-...'

# For persistence (add to ~/.bashrc or ~/.zshrc)
echo 'export NIXTLA_API_KEY="sk-proj-..."' >> ~/.bashrc
source ~/.bashrc
```

**For production** (use secret manager):
```bash
# AWS
export NIXTLA_API_KEY=$(aws secretsmanager get-secret-value --secret-id nixtla-api-key --query SecretString --output text)

# GCP
export NIXTLA_API_KEY=$(gcloud secrets versions access latest --secret=nixtla-api-key)
```

### Issue 2: "Skills not showing up in Claude Code"

**Symptom**: User says "I installed skills but Claude doesn't see them"

**Solution**:
1. Verify installation: `ls .claude/skills/` (should show skill directories)
2. Restart Claude Code
3. Check you're in the correct project directory (skills are per-project)
4. Check `.claude/skills.json` exists (installer should create this)

### Issue 3: "Permission denied" when installing

**Symptom**: `pip install nixtla-claude-skills-installer` fails with permission error

**Solution**:
```bash
# Don't use sudo! Use a virtual environment instead
python3 -m venv venv
source venv/bin/activate
pip install nixtla-claude-skills-installer
```

### Issue 4: "API rate limit exceeded"

**Symptom**: Forecasts fail with `429 Too Many Requests`

**Solution**:
1. Check Nixtla dashboard for rate limits (may be tier-dependent)
2. Implement request throttling in code:
   ```python
   import time
   for series in series_list:
       forecast = client.forecast(series)
       time.sleep(1)  # Rate limit: 1 req/sec
   ```
3. Consider upgrading Nixtla plan if hitting limits frequently
4. Use usage-optimizer skill to batch requests (reduces API calls)

### Issue 5: "Forecast quality suddenly degraded"

**Symptom**: Production forecasts used to be accurate, now they're bad

**Solution**:
1. Check for **data drift** (distribution change):
   ```python
   # Skills generate monitoring code
   python pipelines/monitoring.py --check-drift
   ```

2. Check for **data quality issues** (missing values, outliers):
   ```python
   # Validate data
   python data_processing/nixtla_mapper.py --validate
   ```

3. Check **API status** (Nixtla outage or maintenance):
   - Visit status.nixtla.io
   - Check error logs for API errors

4. **Re-run backtests** to confirm issue:
   ```python
   python forecasting/experiments.py --backtest
   ```

### Issue 6: "High API costs"

**Symptom**: Nixtla bill is higher than expected

**Solution**:
1. **Audit usage** with usage-optimizer skill:
   ```
   "Use nixtla-usage-optimizer to analyze our costs"
   ```

2. **Implement routing strategy** (use baselines for simple forecasts):
   ```python
   def choose_model(complexity):
       if complexity == "simple":
           return statsforecast_model()  # Free
       else:
           return timegpt_model()  # Paid
   ```

3. **Batch requests** (1 API call for 100 series vs 100 API calls):
   ```python
   # Before (inefficient)
   for series in series_list:
       forecast = client.forecast(series)

   # After (batched - 40% cost reduction)
   all_forecasts = client.forecast(all_series_combined)
   ```

4. **Set up cost alerts** in Nixtla dashboard

## Part 6: When to Call for Backup (1 minute)

Here's when to escalate issues:

**Escalate to Nixtla Support** (support@nixtla.io) when:
- API is down or returning errors
- Billing issues or unexpected charges
- Questions about TimeGPT features or accuracy
- Data privacy or compliance questions

**Escalate to Skills Maintainer** (skills repo issues) when:
- Skills generate invalid code
- Skills have bugs or unexpected behavior
- Feature requests for new skills
- Documentation is unclear

**Escalate to Claude Code Support** (claude.ai/support) when:
- Claude Code crashes or freezes
- Skills not loading despite correct installation
- General Claude Code questions

## Summary: The 5 Things to Remember

If you forget everything else, remember these 5 things:

1. **Skills are just files** (`.claude/skills/` directories with SKILL.md files)
2. **Per-project, not global** (each project needs its own skill installation)
3. **API keys must be secure** (use env vars, secret managers, never commit to git)
4. **Monitor costs** (use usage-optimizer skill, set up alerts)
5. **Skills generate code, don't execute it** (users run the generated code manually or via CI/CD)
